# Фиксированный URL веб-приложения для iPad (продуктовый стенд)

Цель: на стенде участник и оператор всегда открывают одно и то же **полное** (origin) — например `https://kiosk.example.local` или `http://10.131.125.69` — без «случайного» IP соседнего интерфейса.

## 1. Что важно в этом проекте

- Фронт ходит в API по относительному пути **`/api`** — удобно отдавать фронт и API **с одного хоста и порта** (reverse proxy), тогда браузер не упирается в CORS между разными origin.
- Если фронт и API на **разных** origin (другой хост или порт), на бэкенде в **production** нужен **`FRONTEND_ORIGIN`** — точное совпадение с тем URL, который вводится в Safari на iPad (схема + хост + порт).

```text
FRONTEND_ORIGIN=https://kiosk.example.local
# или
FRONTEND_ORIGIN=http://10.131.125.69:8080
```

Без этого API может отвечать с ошибкой CORS в продакшене.

Переменные окружения бэкенда см. `apps/backend/.env.example`, точка входа CORS: `apps/backend/src/index.ts`.

## 2. «Жёсткий» адрес: варианты

### Вариант A — один IP (часто на внутренней сети)

1. На **сервере/ПК**, где крутится приложение, задать **статический IPv4** или **резервацию в DHCP** на нужный адрес (например `10.131.125.69`).
2. Убедиться, что iPad в **той же сети** (или есть маршрут до этого сегмента).
3. Открыть в Safari: `http://10.131.125.69:<порт>` (порт — тот, на котором слушает nginx или Node; для HTTPS обычно `443`).

Минус: IP может смениться, если не закрепить в DHCP/ОС.

### Вариант B — имя хоста (предпочтительно для стенда)

1. Ввести запись **DNS** (внутренний DNS или `hosts` на iPad *неудобно для массового использования*) — например `kiosk.stand.local` → нужный IP.
2. Раздавать фронт + API за **reverse proxy** (nginx, Caddy и т.д.) на `https://kiosk.stand.local` (или HTTP только в изолированной сети).
3. **`FRONTEND_ORIGIN`** = тот же URL, что вводит пользователь в браузере (включая `https://` и без лишнего слэша в конце, если так настроен CORS).

Плюс: стабильная закладка на iPad, смена IP только на стороне DNS.

### Вариант C — один origin через reverse proxy

Типичная схема:

- Браузер → `https://kiosk.stand.local` (порт 443).
- Nginx: статика фронта (build из `apps/frontend`) + `location /api { proxy_pass http://127.0.0.1:3001; }`.
- Бэкенд слушает `127.0.0.1:3001`, наружу не обязан быть открыт.

Тогда **`FRONTEND_ORIGIN=https://kiosk.stand.local`** (или с явным порту, если не 443).

## 3. iPad

1. Подключить к сети стенда.
2. В Safari открыть **канонический** URL один раз, проверить работу.
3. **Поделиться → На экран «Домой»** — ярлык на домашний экран (опционально).
4. Если используется self-signed HTTPS — может понадобиться один раз принять сертификат в настройках/при открытии.

## 4. Чеклист для разработчика стенда

| Шаг | Действие |
|-----|----------|
| 1 | Зафиксировать **один** URL для киоска (IP или DNS-имя + схема и порт). |
| 2 | Настроить **статический IP** или **DNS** на этот хост. |
| 3 | Собрать фронт (`npm run build` в frontend), отдать статику + прокси `/api` одним origin. |
| 4 | Выставить **`NODE_ENV=production`** и **`FRONTEND_ORIGIN`** = точный URL из Safari. |
| 5 | Проверить с iPad: главная, регистрация, старт забега, `/api/health` через тот же хост. |

## 5. Dev vs prod

- В **разработке** Vite может быть привязан к конкретному IP (см. `apps/frontend/vite.config.ts`, переменные `VITE_DEV_HOST`, `VITE_DEV_PORT`) — это только для локального/stage dev, не для продуктового стенда.
- На **продуктовом стенде** обычно нет Vite: отдаётся **собранный** статический билд + бэкенд за nginx/systemd.

## 6. Если что-то не открывается

Сначала проверить **сеть и файрвол** (доступность порта с iPad), затем **CORS** (`FRONTEND_ORIGIN`), затем что фронт реально отдаётся по тому URL, который в закладке.

Ограничений по **IP клиента** в коде приложения нет (см. обсуждение в репозитории); блокировки чаще на стороне инфраструктуры.
